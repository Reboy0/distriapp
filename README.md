# Distriapp — B2B «Дистриб'ютор ↔ Магазин»

Монорепозиторій P0-версії системи. Повний функціональний опис —
`docs/funktsionalni-vymohy-dlya-rozrobky.md`; стислий домен/API-контракт, узгоджений між
бекендом і фронтендами, — `docs/CONTRACT.md`; тестові облікові дані — `docs/TEST_CREDENTIALS.md`;
як задеплоїти на Railway + Vercel — `docs/DEPLOYMENT.md`.

## Компоненти

| Каталог | Стек | Призначення |
| --- | --- | --- |
| `backend/` | FastAPI + SQLAlchemy 2 (async) + PostgreSQL + Alembic | API, бізнес-логіка, інтеграція з 1С (абстрактний інтерфейс) |
| `apps/distributor-web/` | Next.js 15 + TypeScript + Tailwind (:3000) | Веб-панель дистриб'ютора |
| `apps/admin-web/` | Next.js 15 + TypeScript + Tailwind (:3001) | Веб-панель адміна платформи |
| `apps/client-web/` | Next.js 15 + TypeScript + Tailwind (:3002) | Кабінет клієнта (представника магазину) — сайт, мобільно-орієнтований |
| `apps/mobile/` | Expo (React Native) + TypeScript + Expo Router | Мобільний застосунок клієнта (iOS/Android) — залишений як є, `client-web` не заміняє його в коді, а існує паралельно |

`apps/client-web` і `apps/mobile` реалізують **той самий** клієнтський функціонал (каталог,
кошик, точки, замовлення, чат) проти одного бекенду — оберіть, який канал розвивати далі.

## Швидкий старт

```bash
# Бекенд
cd backend
python3 -m venv .venv && source .venv/bin/activate
pip install -r requirements.txt
cp .env.example .env
docker-compose up -d postgres   # з кореня репозиторію; якщо порт 5432 зайнятий іншим проєктом —
                                 # див. docker-compose.yml (тут використано 5434)
alembic upgrade head            # міграція вже є в alembic/versions
uvicorn app.main:app --reload   # http://localhost:8000/docs

# Перший адмін (немає самореєстрації — AUTH-1)
python -m scripts.seed_admin admin@example.com yourpassword

# Веб-панелі (з кореня репозиторію)
pnpm install
pnpm dev   # turborepo паралельно піднімає distributor-web / admin-web / client-web

# Мобільний застосунок — окремо, інтерактивний процес
cd apps/mobile && npx expo start
```

## Стан на зараз

Усе нижче **перевірено наживо** (реальний Postgres + бекенд + curl/Playwright для чотирьох
екранних застосунків), а не лише типізацією/лінтом:

- **Бекенд**: повна модель даних під P0-вимоги, бізнес-правила (перевизначення ціни на точку,
  статуси наявності, блокування точки + відстрочка, життєвий цикл замовлення з миттєвою передачею
  в 1С), JWT-автентифікація чотирьох ролей (admin/distributor/client), абстрактний конектор 1С +
  заглушка. Наскрізний сценарій пройдено через реальний API: онбординг → запрошення → реєстрація
  клієнта → точка → каталог → замовлення → блокування → відстрочка знімає блокування.
- **distributor-web / admin-web**: усі P0-екрани підключені до реального бекенду й перевірені
  в браузері (Playwright): точки з підсвіткою боржників, ручна ціна на точку (перегляд+встановлення
  +скидання), відстрочка (перегляд+редагування), замовлення (список/деталі/скасування з коментарем/
  зміна оплати), запрошення, чат з карткою товару, ручна синхронізація з 1С, CRUD дистриб'юторів.
- **client-web** (новий, за рішенням замовника — клієнтський інтерфейс теж сайт, а не лише
  мобільний застосунок): реєстрація за запрошенням, вхід, точки (список/додавання). Дашборд точки
  (`/points/[id]`) показує останні 3 замовлення й кнопку в каталог; каталог і кошик — окрема
  сторінка (`/points/[id]/catalog`), «Замовити» без екрана підтвердження. Мобільно-орієнтований
  layout (нижня навігація: Точки/Замовлення/Чат) з фіксованими панелями, вирівняними під safe-area
  мобільних пристроїв (`--nav-h` + `env(safe-area-inset-*)` — жодних захардкоджених офсетів).
  Кошик зберігається в localStorage окремо на кожну точку. Повний цикл перевірено в браузері проти
  реального бекенду.
- **Живі оновлення (WebSocket)**: `WS /api/v1/ws?token=...` — чат-повідомлення та зміни замовлень
  (статус, скасування, посилання на оплату) прилітають в обидва боки (client-web і
  distributor-web) без перезавантаження сторінки чи polling. Перевірено наживо: дистриб'ютор
  скасовує/оплачує замовлення чи пише в чат — клієнт бачить це миттєво, і навпаки.
- **Посилання на оплату**: не інтеграція з платіжною системою — дистриб'ютор вручну вставляє URL
  (зі свого інвойсингу/іншого сервісу) на сторінці замовлення; клієнт бачить кнопку «Оплатити» на
  неоплаченому замовленні. Перевірено наживо в обох панелях.
- **mobile**: типізація чиста (`tsc --noEmit`), структура екранів під ту саму бізнес-логіку (без
  описаних вище нових фіч — вони поки лише в client-web); Expo web-прев'ю в цьому середовищі
  впирається в ланцюжок недооголошених peer-залежностей специфічний для web-таргету (не блокер
  для реальної платформи iOS/Android).
- **Не реалізовано**: реальний конектор 1С (Q1-Q3 не мають відповіді замовника), справжній
  планувальник для щоденного імпорту каталогу й синхронізації оплат 3×/день (є ручна кнопка +
  сервісні функції), push-сповіщення (новинки), деактивація клієнта (AUTH-6, P1), моніторинг
  інтеграцій в адмінці (ADM-2, P1) — заглушка, акти звірки (DOC-1/2, P1). Автозамовлення за
  розкладом було реалізовано і потім прибрано за рішенням замовника.

## Відомі особливості монорепо (pnpm + Next.js + Expo)

- Корінь має `pnpm.overrides` для `@types/react`/`@types/react-dom` — без цього TypeScript
  плутає дві версії React-типів (18.3.x для сайтів, 18.2.x для Expo) і Next.js падає з
  незрозумілою помилкою `bigint is not assignable to ReactNode` на кожному layout-файлі.
- `apps/mobile/metro.config.js` явно налаштований під pnpm workspace (`watchFolders`,
  `nodeModulesPaths`, `unstable_enableSymlinks`) — без цього Metro не бачить пакети за
  symlink-структурою pnpm.

## Наступні кроки

1. Визначитись остаточно: `client-web` чи `apps/mobile` (чи обидва) — і викреслити зайве.
2. Закрити технічні питання Q1-Q4 з `docs/funktsionalni-vymohy-dlya-rozrobky.md` (сесія з
   1С-спеціалістом першого дистриб'ютора) і реалізувати конкретний `OneCConnector`.
3. Замінити `app/scheduler.py` (in-process asyncio-цикл, 30с — зручно для дев-середовища) на
   справжній планувальник (Celery beat / APScheduler / cron) і додати туди ж щоденний імпорт
   каталогу й синхронізацію оплат 3×/день (`services/orders.list_orders_pending_retry` вже готовий
   до підключення retry-черги).
4. WebSocket зараз single-process (in-memory `ConnectionManager`) — для кількох воркерів бекенду
   знадобиться спільний брокер (Redis pub/sub).
5. Push-сповіщення, деактивація клієнта, моніторинг інтеграцій для адміна.
