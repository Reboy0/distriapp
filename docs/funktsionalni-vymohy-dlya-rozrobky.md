# Функціональні вимоги для розробки: B2B-система «Дистриб'ютор ↔ Магазин»

**Версія:** 1.2 (для передачі команді розробки) 

## 1\. Огляд

Система замінює торгового менеджера дистриб'ютора: клієнти (магазини) замовляють продукцію через мобільний застосунок з індивідуальними цінами, замовлення автоматично потрапляють у 1С дистриб'ютора, система контролює оплати та доносить новинки.

**Компоненти та платформи:**

1. Застосунок представника магазину (клієнта) — **мобільний, iOS \+ Android, кросплатформенна розробка**  
2. Панель дистриб'ютора — **web**  
3. Панель адміна платформи — **web**  
4. Конектор до 1С (окреме підключення для кожного дистриб'ютора)

## 2\. Терміни

| Термін | Значення |
| :---- | :---- |
| **Дистриб'ютор** | Компанія-постачальник. Тенант системи, має власну 1С. |
| **Клієнт** | Представник магазину — користувач мобільного застосунку. Належить одному дистриб'ютору. |
| **Торгова точка** | Конкретний магазин клієнта. Один клієнт має 1..N точок. Замовлення, ціни та борг прив'язані до точки. |
| **Замовлення** | Створюється клієнтом для однієї точки. Після створення автоматично передається в 1С як непроведений документ. |

## 3\. Ключові рішення (зафіксовано із замовником)

- Ціни відображаються **лише з ПДВ** — усюди (каталог, кошик, історія).  
- Базова ціна тягнеться з 1С; дистриб'ютор може **вручну перевизначити ціну для кожної торгової точки окремо**. Ручна ціна не перезаписується імпортом; сповіщень про розбіжність з базовою ціною не потрібно.  
- **Жодних кроків підтвердження**: ні клієнтом (окремого екрана підтвердження немає — кнопка «Замовити» в кошику одразу відправляє замовлення), ні дистриб'ютором. Замовлення летить у 1С одразу після натискання «Замовити».  
- Дистриб'ютор може **скасувати** замовлення. У 1С при цьому нічого автоматично не змінюється — непроведений документ видаляє оператор 1С вручну.  
- Статус «не оплачено» **не ставиться автоматично при створенні замовлення** — він з'являється після доставки: з 1С (синхронізація) або вручну дистриб'ютором. Лише замовлення з явним статусом «не оплачено» блокує точку.  
- Борг прив'язаний **до конкретної торгової точки**: блокується лише точка з неоплаченим замовленням, інші точки клієнта працюють.  
- **Відстрочка ставиться на точку** (не на замовлення): вручну, з датою; до дати точка не блокується; після дати за наявності неоплачених замовлень точка стає «боржником» (червона підсвітка \+ блок).  
- Ручна позначка «оплачено» **остаточна**: наступні синхронізації з 1С її не скасовують.  
- Синхронізація оплат — **тричі на день: 9:00, 15:00, 20:00 за Києвом**.  
- **Кратність замовлення поки не реалізуємо** — клієнт вводить довільну кількість.  
- **Друк накладної — на стороні 1С**, поза системою; в апці цієї опції не буде.  
- Поріг «закінчується» встановлюється **вручну для кожного товару окремо**. Клієнт бачить лише статус «є / закінчується / немає», без точних залишків.  
- Час доставки не відображається. Push-нагадувань про оплату немає.

## 4\. Статусні моделі

### 4.1. Життєвий цикл замовлення

створене клієнтом (кнопка «Замовити» в кошику, без екрана підтвердження)

   → передається в 1С            (автоматично, одразу)

   → \[очікує передачі\]           (якщо 1С недоступна; авто-повтори)

   → передане в 1С               (непроведений документ створено)

   → скасоване                   (дистриб'ютором, з коментарем;

                                  документ у 1С видаляє оператор вручну)

### 4.2. Статус оплати замовлення

| Статус | Як встановлюється | Ефект |
| :---- | :---- | :---- |
| **Створено** (без платіжного статусу) | Початковий стан. Автоматично «не оплачено» НЕ ставиться | Не блокує нічого — клієнт може створювати наступні замовлення для цієї точки |
| **Не оплачено** | Після доставки: з 1С (синхронізація 3×/день) або вручну дистриб'ютором | Блокує точку (якщо на точці немає активної відстрочки), червона підсвітка |
| **Оплачено** | З 1С при синхронізації, або вручну дистриб'ютором. **Ручна позначка остаточна** — наступні синхронізації з 1С її не скасовують | Знімає блокування точки одразу |

### 4.3. Відстрочка та блокування точки

Відстрочка — атрибут **торгової точки**, не замовлення:

- Дистриб'ютор вручну ставить точці відстрочку із зазначенням дати.  
- Поки дата відстрочки не минула, точка **не блокується і не підсвічується**, навіть якщо має замовлення «не оплачено».  
- Після дати відстрочки, якщо на точці залишаються неоплачені замовлення, точка автоматично стає **«боржником»**: червона підсвітка \+ блокування. Перехід відбувається о 00:00 наступного дня після дати відстрочки.

**Правило блокування:** клієнт не може створити замовлення для точки, якщо (точка має ≥1 замовлення у статусі «не оплачено») І (точка не має активної відстрочки). Інші точки того ж клієнта не блокуються.

## 5\. Функціональні вимоги

Пріоритети: **P0** — перша версія, **P1** — наступна версія, **P2** — майбутнє.

### 5.1. Автентифікація та онбординг (AUTH)

| ID | Пріоритет | Вимога |
| :---- | :---- | :---- |
| AUTH-1 | P0 | Адмін створює акаунт дистриб'ютора: назва, контакти, параметри підключення до 1С. |
| AUTH-2 | P0 | Дистриб'ютор генерує персональне запрошення для клієнта (унікальне посилання/код). **Термін дії запрошення — 7 днів**, після чого воно протерміновується; дистриб'ютор може згенерувати нове. Бачить список запрошень зі статусами: активне / використане / протерміноване. |
| AUTH-3 | P0 | Реєстрація клієнта — виключно за запрошенням. Відкритої реєстрації немає. Після реєстрації клієнт автоматично прив'язаний до дистриб'ютора, що видав запрошення. |
| AUTH-4 | P0 | Ізоляція тенантів: користувач бачить лише дані свого дистриб'ютора. Крос-тенантний доступ виключений на рівні API. |
| AUTH-5 | P0 | Кількість пристроїв на один акаунт клієнта **не обмежується** — одночасні активні сесії з кількох пристроїв дозволені. |
| AUTH-6 | P1 | Деактивація клієнта дистриб'ютором: вхід заборонено, історія зберігається. |

### 5.2. Каталог, ціни, наявність (CAT)

| ID | Пріоритет | Вимога |
| :---- | :---- | :---- |
| CAT-1 | P0 | Номенклатура та залишки імпортуються з 1С дистриб'ютора щодня у налаштований час. |
| CAT-2 | P0 | Базова ціна товару тягнеться з 1С. Дистриб'ютор може вручну перевизначити ціну **для кожної торгової точки окремо**. Ручна ціна має пріоритет над базовою і **не перезаписується** наступними імпортами з 1С. |
| CAT-3 | P0 | Усі ціни відображаються **лише з ПДВ** — у каталозі, кошику, картці замовлення, історії. Ціна без ПДВ ніде в UI не показується. |
| CAT-4 | — | ~~Кратність замовлення~~ — **відкладено рішенням замовника**. У v1 клієнт вводить довільну кількість в одиниці виміру товару. Архітектурно закласти можливість додати валідацію кратності пізніше. |
| CAT-5 | P0 | Статус наявності для клієнта: «є» (залишок \> поріг), «закінчується» (0 \< залишок ≤ поріг), «немає» (залишок \= 0). Поріг встановлюється дистриб'ютором **вручну для кожного товару окремо**. Якщо поріг не задано — показуються лише «є / немає». Точні залишки клієнту не показуються ніде. |
| CAT-6 | P0 | Новинки: дистриб'ютор позначає товар як новинку → бейдж у каталозі \+ push-сповіщення всім активним клієнтам дистриб'ютора. |
| CAT-7 | P1 | Рекомендації в кошику: «з цим також замовляють». |

**Acceptance criteria (CAT-2):**

- [ ] Given для точки задана ручна ціна 120 грн, When нічний імпорт приносить з 1С базову ціну 110 грн, Then для точки залишається 120 грн, а базова ціна оновлюється для точок без ручної ціни  
- [ ] Дистриб'ютор бачить, для яких точок ціна ручна, а для яких — базова з 1С, і може скинути ручну ціну до базової

### 5.3. Торгові точки (POS)

| ID | Пріоритет | Вимога |
| :---- | :---- | :---- |
| POS-1 | P0 | Клієнт бачить список своїх торгових точок. |
| POS-2 | P0 | Клієнт додає нову точку (назва, адреса). Точка активна для замовлень одразу, без підтвердження дистриб'ютором. Для нової точки діють базові ціни з 1С, доки дистриб'ютор не задасть ручні. |
| POS-3 | P0 | Дистриб'ютор бачить усі точки всіх своїх клієнтів. |

### 5.4. Замовлення (ORD)

| ID | Пріоритет | Вимога |
| :---- | :---- | :---- |
| ORD-1 | P0 | Замовлення створюється для однієї конкретної точки. Точка обирається до або під час формування кошика. |
| ORD-2 | P0 | У кошику клієнт вказує кількість в одиниці виміру товару (валідація кратності відкладена — див. CAT-4). Кошик показує позиції, кількість, ціну з ПДВ за одиницю, суму за позицію, загальну суму. Час доставки не відображається. |
| ORD-3 | P0 | Кнопка «Замовити» в кошику одразу створює замовлення — **окремого екрана підтвердження клієнтом немає**. |
| ORD-4 | P0 | Створене замовлення **автоматично** передається в 1С як непроведений документ. Кроку підтвердження дистриб'ютором немає. |
| ORD-5 | P0 | Якщо 1С недоступна в момент передачі — замовлення отримує статус «очікує передачі», система робить автоматичні повтори; після N невдалих спроб дистриб'ютор отримує сповіщення. Замовлення не губиться. |
| ORD-6 | P0 | Нове замовлення **не отримує** платіжного статусу автоматично. Статус «не оплачено» встановлюється після доставки: з 1С при синхронізації або вручну дистриб'ютором. |
| ORD-7 | P0 | **Блокування:** клієнт не може створити замовлення для точки з неоплаченим замовленням, якщо на точці немає активної відстрочки (правило — розділ 4.3). Клієнт бачить пояснення: яка точка заблокована і яке замовлення не оплачене. Інші точки клієнта доступні. |
| ORD-8 | P0 | **Відстрочка на точку:** дистриб'ютор вручну ставить точці відстрочку з датою; може змінити або зняти її. Після дати відстрочки за наявності неоплачених замовлень точка автоматично стає «боржником» (розділ 4.3). |
| ORD-9 | P0 | У панелі дистриб'ютора точки-«боржники» та їхні неоплачені замовлення підсвічуються червоним. Є фільтр за платіжним статусом. |
| ORD-10 | P0 | Дистриб'ютор може вручну змінити платіжний статус замовлення (оплачено / не оплачено). Ручна позначка «оплачено» **остаточна**: знімає блокування одразу і не скасовується наступними синхронізаціями з 1С. |
| ORD-11 | P0 | **Скасування:** дистриб'ютор може скасувати замовлення з коментарем. Клієнт бачить статус «скасоване» та причину. У 1С система нічого не змінює — непроведений документ видаляє оператор 1С вручну. |
| ORD-12 | P0 | Історія замовлень для обох сторін зі статусами життєвого циклу та оплати, з фільтром за точкою. |

**Acceptance criteria (ORD-3/4, ORD-7, ORD-8):**

- [ ] Given клієнт натиснув «Замовити» в кошику, When передача в 1С успішна, Then у 1С існує непроведений документ, а замовлення в апці має статус «передане в 1С» — без жодних додаткових підтверджень  
- [ ] Given щойно створене замовлення без платіжного статусу, When клієнт створює ще одне замовлення для цієї ж точки, Then створення дозволено  
- [ ] Given точка А має замовлення «не оплачено» і не має відстрочки, When клієнт відкриває створення замовлення, Then точка А недоступна з поясненням, а точка Б того ж клієнта — доступна  
- [ ] Given точка А має замовлення «не оплачено» і відстрочку до 15.08, When 14.08 клієнт створює замовлення для точки А, Then створення дозволено  
- [ ] Given та сама точка, When настає 16.08 і замовлення не оплачене, Then точка автоматично «боржник»: червона підсвітка \+ блокування  
- [ ] Given дистриб'ютор вручну позначив замовлення «оплачено», When синхронізація з 1С повертає «не оплачено», Then статус залишається «оплачено»

| INT-6 | P1 | Журнал синхронізацій: час останнього успішного обміну, помилки. Видимий дистриб'ютору та адміну. |

## 6\. Non-goals (свідомо поза скоупом)

- Крок підтвердження замовлення — ні клієнтом (окремий екран), ні дистриб'ютором: «Замовити» в кошику одразу шле замовлення в 1С.  
- **Кратність замовлення** — відкладено рішенням замовника; клієнт вводить довільну кількість. *Ризик зафіксовано: початкова проблема «замовив 2 л замість 200» у v1 не закривається.*  
- **Друк видаткової накладної з системи** — друк відбувається на стороні 1С, поза застосунком.  
- Автоматичні дії в 1С при скасуванні замовлення — непроведений документ видаляє оператор 1С вручну.  
- Сповіщення дистриб'ютору про розбіжність ручної ціни точки з оновленою базовою ціною 1С.  
- Орієнтовний час доставки.  
- Push-нагадування про наближення терміну оплати.  
- Онлайн-оплата в застосунку.  
- Проведення документів у 1С (тільки непроведене замовлення).  
- Логістика: маршрути, трекінг доставки.  
- Відкрита реєстрація клієнтів.  
- Відображення цін без ПДВ.

## 7\. Питання для довизначення

Продуктові рішення закриті замовником і внесені в розділи 3–5. Залишаються технічні питання по інтеграції з 1С — замовник підтвердив, що відповіді наразі невідомі й будуть визначені пізніше. **Для команди це означає: конектор 1С проєктувати з абстрактним інтерфейсом обміну, а оцінку по ньому давати з припущеннями.**

| \# | Питання | Статус | Що блокує |
| :---- | :---- | :---- | :---- |
| Q1 | **Конфігурації та версії 1С** цільових дистриб'юторів. | Невідомо. Дія: з'ясувати у перших 2–3 дистриб'юторів. | Оцінку конектора та спосіб обміну (COM-модуль, HTTP-сервіси, файловий обмін) |
| Q2 | **Маппінг сутностей апка ↔ 1С:** ключі зіставлення контрагентів/точок, номенклатури (код, артикул, GUID), замовлень та оплат. Хто заводить точку в 1С, коли клієнт створив її в апці (POS-2)? | Невідомо, замовник визначить пізніше. | Дизайн конектора. До відповіді: точки, створені в апці, зберігати з власним ID і полем для майбутньої прив'язки до 1С |
| Q3 | **Ознака «не оплачено» в 1С:** за яким документом/станом 1С визначає, що замовлення доставлене й не оплачене. | Невідомо, замовник визначить пізніше. | Реалізацію імпорту оплат (INT-3). Логіку апки (ORD-6/ORD-7) можна будувати вже — вона працює зі статусом незалежно від того, як він обчислюється в 1С |
| Q4 | Значення N (кількість невдалих спроб передачі в 1С) та інтервали повторів для ORD-5. | Пропонований дефолт: 5 спроб з експоненційною затримкою, далі алерт дистриб'ютору. | — |

Рекомендований порядок: усі три питання Q1–Q3 закриваються однією сесією з 1С-спеціалістом першого дистриб'ютора — варто провести її до старту розробки конектора, але це не блокує розробку застосунку, панелей і всієї бізнес-логіки.

## 8\. Фазування

1. **Фаза 1 (P0):** онбординг і запрошення, каталог з цінами з ПДВ і ручним перевизначенням по точках, статуси наявності з ручними порогами, новинки з push, кошик і замовлення з автоматичною передачею в 1С (без підтверджень), платіжні статуси та блокування точки, відстрочки на точку, ручна зміна оплати, скасування, чат \+ «коли буде?», дзвінок, історія, синхронізація (каталог щодня, оплати 3×/день) \+ кнопка оновлення.  
2. **Фаза 2 (P1):** рекомендації в кошику, акти звірки за запитом, деактивація клієнтів, журнал синхронізацій, моніторинг для адміна.  
3. **Фаза 3 (P2):** кратність замовлення (якщо повернемось до неї), автоматичні акти звірки, статистика платформи.
